# AI Project Cockpit PRD

## 1. 产品名称

暂定名：**AI Project Cockpit**  
中文名：**AI 项目驾驶舱**

---

## 2. 一句话定位

为 2–20 人 AI-native 小团队设计的项目协作网页，让每个项目的目标、任务、上下文、AI 执行记录和关键决策集中在一个界面中，帮助团队减少沟通成本、减少上下文丢失，并更高效地让 AI 参与项目推进。

---

## 3. 背景与问题

### 3.1 背景

在 AI 时代，小团队的组织形态正在发生变化：

- 团队规模更小，通常是 2–20 人。
- 每个项目通常由 2–3 人推进。
- 成员往往身兼多职，例如产品、运营、增长、内容、技术、数据分析等角色交叉承担。
- AI 不再只是辅助写文案或总结会议，而是逐渐成为参与方案生成、任务拆解、代码实现、数据分析、用户研究和内容生产的“准团队成员”。

传统协作工具，如飞书、Notion、Slack、Linear、Trello 等，分别解决了沟通、文档、任务、项目管理等问题，但它们大多不是围绕“人和 AI 共同推进项目”设计的。

### 3.2 当前痛点

#### 3.2.1 信息分散

一个项目的信息通常分散在：

- 聊天群
- 文档
- 表格
- 会议纪要
- 任务列表
- 代码仓库
- AI 对话记录
- 临时笔记

导致项目成员和 AI 都很难快速理解“这个项目现在到底是什么状态”。

#### 3.2.2 上下文丢失

AI 参与项目时，最大的障碍不是生成能力，而是上下文不完整：

- 不知道项目目标。
- 不知道历史决策。
- 不知道哪些方案已经被否定。
- 不知道当前阻塞点。
- 不知道哪些资料可信。
- 不知道最新进展。

因此团队经常需要反复向 AI 解释背景。

#### 3.2.3 项目状态不透明

小团队虽然沟通成本低，但也容易出现：

- 任务没有明确 owner。
- 下一步行动不清楚。
- 重要决策只存在于聊天记录里。
- 项目推进依赖某个人的脑内记忆。
- 周会或同步会反复确认同样的问题。

#### 3.2.4 AI 工作结果难以沉淀

AI 生成过很多方案、分析、代码、调研和文案，但这些结果往往停留在单次对话中：

- 很难追踪某次 AI 输出是否被采纳。
- 很难知道 AI 基于什么上下文生成。
- 很难复用之前的 AI 推理结果。
- 很难把 AI 输出转化为任务或决策。

## 4. 产品目标

### 4.1 核心目标

构建一个轻量级网页应用，让小团队可以围绕项目进行 AI-native 协作。

它要回答五个核心问题：

1. 这个项目的目标是什么？
2. 这个项目现在进展到哪里？
3. 接下来最重要的任务是什么？
4. AI 可以基于哪些上下文参与工作？
5. 团队做过哪些关键决策，为什么？

### 4.2 MVP 阶段目标

MVP 明确定位为**内部自用工具**，不优先考虑商业化、标准化销售、企业权限、复杂组织管理或通用 SaaS 能力。

第一版的目标不是做一个完美产品，而是做一个团队自己每天愿意打开、觉得顺手、能真实减少项目推进摩擦的内部项目驾驶舱。

MVP 需要验证的核心假设是：

> 如果项目目标、任务、上下文、关键决策和必要的 AI 辅助能力集中在一个驾驶舱中，2–3 人项目组是否可以更少翻聊天记录、更少重复解释背景，并更清楚地知道下一步要做什么？

第一版原则：

- 怎么用得爽，怎么设计。
- 不为了未来商业化提前做复杂抽象。
- 不做大而全协作平台。
- 不替代现有 AI 工具。
- 不强行把 AI 做成主入口。
- 先让项目状态、任务、资料和决策变清楚。

---

## 5. 非目标

MVP 阶段不做以下事情：

- 不做即时通讯。
- 不做复杂权限系统。
- 不做实时多人协作文档编辑。
- 不做完整 OKR 系统。
- 不做复杂甘特图。
- 不做企业审批流。
- 不做完整知识库替代品。
- 不做移动端 App。
- 不做深度组织管理。
- 不做类似飞书的全功能平台。
- 第一版不接入飞书。
- 第一版不做商业化 SaaS 需要的复杂租户、计费、权限和审计。
- 第一版不试图替代 Cursor、ChatGPT、Claude、Manus、Devin、Perplexity 等外部 AI 工具。

## 6. 目标用户

### 6.1 主要用户

#### 小型 AI-native 创业团队

规模：2–20 人  
特征：

- 项目多，团队小。
- 每个人身兼多职。
- 高频使用 AI。
- 需要快速试错。
- 不希望被重型项目管理工具拖慢。

#### 超级个体工作室

规模：1–5 人  
特征：

- 项目以客户、内容、产品、增长实验为单位。
- 大量工作由 AI 辅助完成。
- 需要管理多个并行项目。
- 需要清晰知道每个项目的下一步。

#### 产品/增长/内容/技术混合小队

规模：2–3 人  
特征：

- 项目推进速度快。
- 需要频繁做决策。
- 需要 AI 帮助做调研、写方案、写代码、生成内容、分析数据。

### 6.2 次要用户

- 小型咨询团队
- 独立开发者团队
- AI 自动化服务团队
- 自媒体商业化团队
- 内部创新项目小组

## 7. 核心使用场景

### 7.1 场景一：创建一个新项目

用户想启动一个新项目，例如“AI 协作工具 MVP”。

用户希望快速记录：

- 项目目标
- 项目背景
- 项目成员
- 当前阶段
- 本周重点
- 初始任务
- 相关资料

系统应帮助用户把模糊想法转化为结构化项目。

### 7.2 场景二：每天打开项目驾驶舱看状态

用户每天打开某个项目页面，希望立刻看到：

- 项目当前状态
- 本周最重要的 3 件事
- 当前阻塞点
- 待处理任务
- 最近 AI 运行结果
- 最近关键决策

目标是让用户不用翻聊天记录，也能知道项目该怎么推进。

### 7.3 场景三：让 AI 基于项目上下文工作

用户希望 AI 不只是回答一个孤立问题，而是基于当前项目的完整上下文执行任务，例如：

- 总结项目当前进展。
- 找出当前最大阻塞点。
- 生成下一步任务。
- 根据上下文写 PRD。
- 根据用户反馈生成改版方案。
- 根据决策日志判断某个方案是否符合方向。

### 7.4 场景四：记录关键决策

团队做了一个重要判断，例如：

- 先做网页，不做移动端。
- 先做项目驾驶舱，不做 IM。
- 暂时不接飞书 API。
- MVP 只支持单团队使用。

这些决策需要被记录下来，并且说明原因。未来 AI 或新成员可以理解为什么当时这么选。

### 7.5 场景五：把 AI 输出转成任务或决策

AI 生成了一份建议，例如“下一步应该做用户访谈、任务模型设计、项目页面原型”。

用户可以从 AI 输出中直接创建：

- 任务
- 决策
- 项目上下文
- 项目备注

这样 AI 输出不只是停留在对话里，而是进入项目系统。

## 8. 产品形态

### 8.1 总体布局

产品是一个网页应用，核心界面采用左右分栏布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部导航：团队 / 搜索 / 新建 / 设置                           │
├───────────────┬──────────────────────────────────────────────┤
│ 左侧项目列表   │ 右侧项目驾驶舱                                │
│               │                                              │
│ - All Projects│  项目标题 / 状态 / 成员 / 阶段                  │
│ - Active      │                                              │
│ - Paused      │  本周重点                                     │
│ - Shipped     │  任务列表                                     │
│               │  上下文区                                     │
│ Project A     │  AI 工作区                                    │

│ Project C     │                                              │
└───────────────┴──────────────────────────────────────────────┘
```

### 8.2 左侧：项目导航

左侧主要用于快速切换项目。

包含：

- 项目搜索
- 新建项目按钮
- 项目状态筛选
- 项目列表
- 每个项目的简要状态

项目列表项展示：

- 项目名称
- 当前阶段
- 最近更新时间
- 是否有阻塞
- 今日/本周待办数量

### 8.3 右侧：项目驾驶舱

右侧是核心工作区，包含项目的完整操作界面。

建议分为以下模块：

1. 项目头部
2. 本周重点
3. 任务区
4. 上下文区
5. AI 工作区
6. 决策日志
7. 活动记录

## 9. 信息架构

### 9.1 核心对象

MVP 包含五个核心对象：

1. Project
2. Task
3. Context
4. AI Run
5. Decision

### 9.2 Project 项目

Project 是最高层级对象。

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 项目 ID |
| name | string | 项目名称 |
| one_liner | string | 一句话目标 |
| description | text | 项目详细背景 |
| status | enum | active / paused / shipped / archived |
| stage | enum | exploring / planning / building / testing / launched |
| owner_id | string | 项目负责人 |
| member_ids | array | 项目成员 |
| priority | enum | low / medium / high |
| weekly_focus | array | 本周最重要的 1–3 件事 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 9.3 Task 任务

Task 代表具体可执行事项。

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 任务 ID |
| project_id | string | 所属项目 |
| title | string | 任务标题 |
| description | text | 任务说明 |
| owner_id | string | 负责人 |
| status | enum | todo / doing / blocked / done / canceled |
| priority | enum | low / medium / high |
| due_date | date | 截止日期 |
| source | enum | manual / ai_generated |
| related_ai_run_id | string | 关联 AI Run |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 9.4 Context 上下文

Context 是项目资料、背景、链接、会议纪要、用户反馈等信息的统一载体。

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 上下文 ID |
| project_id | string | 所属项目 |
| title | string | 标题 |
| type | enum | note / doc / link / meeting / user_feedback / research / file |
| content | text | 正文内容 |
| url | string | 外部链接，可选 |
| source | string | 来源 |
| importance | enum | low / medium / high |
| created_by | string | 创建人 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 9.5 AI Record / AI Note

第一版暂不把 AI 记录定义成复杂的“AI 执行记录”。因为团队实际会使用很多外部 AI 工具完成工作，例如 ChatGPT、Claude、Cursor、Perplexity、Manus、Devin 或其他专门工具。

因此，MVP 中更合理的定义是：

> AI Record 不是记录 AI 在系统内完整执行了什么，而是记录“某次外部或内部 AI 工作产出了什么有价值的结果，以及这个结果如何影响当前项目”。

它更像一张“AI 工作结果卡片”，而不是一个自动化执行日志。

典型场景：

- 用户在 ChatGPT 里生成了一版 PRD，把核心结论粘贴进项目。
- 用户在 Cursor 里让 AI 改了一段代码，把实现说明记录进项目。
- 用户用 Perplexity 做了调研，把结论和链接保存进项目。
- 用户让 Claude 做竞品分析，把摘要保存为项目上下文。
- 用户在本工具内调用 AI 生成了下一步任务。

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | AI Record ID |
| project_id | string | 所属项目 |
| title | string | 标题 |
| tool | string | 使用的 AI 工具，如 ChatGPT / Claude / Cursor / Other |
| input_summary | text | 输入或需求摘要，可选 |
| output_summary | text | AI 输出的核心结论 |
| raw_output | text | 原始输出，可选 |
| value | enum | insight / task_suggestion / draft / code / research / decision_support |
| status | enum | saved / used / rejected / converted |
| converted_to | json | 是否转成任务、上下文或决策 |
| created_by | string | 记录人 |
| created_at | datetime | 创建时间 |

第一版不强制用户记录完整 prompt 和完整输出，只鼓励保存有价值的结论。

### 9.6 Decision 决策

Decision 记录项目关键判断。

字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 决策 ID |
| project_id | string | 所属项目 |
| title | string | 决策标题 |
| decision | text | 决策内容 |
| rationale | text | 决策原因 |
| alternatives | text | 被考虑但未选择的方案 |
| impact | text | 影响范围 |
| decided_by | array | 决策人 |
| created_at | datetime | 创建时间 |

- 项目背景
- 状态
- 阶段
- 成员
- 本周重点

### 10.1.3 项目列表

用户可以在左侧查看所有项目。

支持：

- 按状态筛选
- 按关键词搜索
- 按更新时间排序
- 标记阻塞项目

## 10.2 项目驾驶舱

### 10.2.1 项目头部

展示：

- 项目名称
- 一句话目标
- 状态
- 阶段
- owner
- 成员
- 最近更新时间

### 10.2.2 本周重点

展示当前项目本周最重要的 1–3 件事。

用户可以：

- 添加重点
- 编辑重点
- 删除重点
- 将重点转为任务

### 10.2.3 项目摘要

系统展示一个项目摘要区域，内容包括：

- 当前阶段
- 当前进展

- 最近完成事项

该摘要可以由 AI 生成，也可以手动编辑。

字段：

- owner

任务状态包括：

- Todo

- Done

AI 输出中可以识别建议任务，用户可以一键转为任务。

## 10.4 上下文管理

### 10.4.1 新增上下文

用户可以添加不同类型的上下文：

- 普通笔记
- 文档内容
- 链接
- 会议纪要
- 用户反馈
- 调研资料

### 10.4.2 上下文列表

展示项目相关上下文。

支持：

- 按类型筛选
- 按重要性筛选
- 搜索
- 标记重要上下文

### 10.4.3 AI 使用上下文

当用户在 AI 工作区发起任务时，系统默认可以引用当前项目的上下文。

用户可以选择：

- 使用全部上下文
- 只使用重要上下文
- 手动选择部分上下文

MVP 阶段可以先实现“使用全部上下文摘要”。

## 10.5 AI 辅助模块

第一版中，AI 不是产品主入口，而是项目驾驶舱里的辅助模块。

产品不试图替代外部 AI 工具，而是做两件事：

1. 让用户可以在当前项目上下文里快速调用 AI 做一些轻量工作。
2. 让用户可以把外部 AI 工具产生的有价值结果沉淀回项目。

### 10.5.1 AI 辅助入口

每个项目页面保留一个 AI 辅助模块。

用户可以输入自然语言请求，例如：

- 总结当前项目。
- 帮我找出阻塞点。
- 基于上下文生成下一步任务。
- 根据项目资料写一版说明。
- 帮我整理这些用户反馈。

### 10.5.2 快捷指令

MVP 提供几个默认快捷指令：

1. 总结当前项目状态
2. 生成下一步任务
3. 找出当前阻塞点
4. 生成本周计划
5. 整理上下文摘要

### 10.5.3 AI 结果记录

用户可以保存两类 AI 结果：

#### 系统内 AI 结果

用户在本工具中调用 AI 后，系统保存本次输出。

#### 外部 AI 结果

用户在其他 AI 工具中完成工作后，可以把重要结果粘贴进来，保存为 AI Record。

外部 AI 结果不要求完整记录，只需要保存对项目有价值的部分。

### 10.5.4 AI 输出操作

用户可以对 AI 输出执行操作：

- 保存为上下文
- 转为任务
- 转为决策
- 标记已使用
- 标记暂不采用

## 10.6 决策日志

### 10.6.1 创建决策

用户可以手动记录决策。

字段：

- 决策标题
- 决策内容
- 决策原因
- 替代方案
- 影响范围
- 决策人

### 10.6.2 AI 生成决策草稿

## 10.7 搜索

MVP 需要一个基础搜索能力。

搜索范围：

- 项目名称
- 项目描述
- 任务标题
- 上下文标题和正文

## 15. AI 能力设计

### 15.1 上下文组装

每次调用 AI 时，系统应组装以下内容：

- 项目名称
- 一句话目标
- 项目背景
- 当前阶段
- 本周重点
- 未完成任务
- 阻塞任务
- 重要上下文
- 最近决策
- 最近 AI Run 摘要

### 15.2 默认系统提示词方向

AI 应被设定为项目协作助手，而不是普通聊天助手。

它需要：

- 理解项目目标。
- 基于上下文回答。
- 明确区分事实、推测和建议。
- 输出可执行任务。
- 标注需要人类确认的地方。
- 避免编造项目中不存在的信息。

### 15.3 输出格式

AI 输出尽量结构化，例如：

```markdown
## 当前判断

## 主要依据

## 建议下一步

## 可创建任务

## 需要人类确认的问题
```

### 15.4 快捷指令 Prompt 草案

#### 总结当前项目状态

```text
请基于当前项目上下文，总结这个项目的当前状态。
请包括：
1. 当前目标
2. 当前阶段
3. 已完成事项
4. 正在推进事项
5. 阻塞点
6. 下一步建议
```

#### 生成下一步任务

```text
请基于当前项目上下文，生成接下来最重要的 3–5 个任务。
每个任务请包括：任务标题、任务说明、建议 owner、优先级、原因。
```

#### 找出当前阻塞点

```text
请基于当前项目上下文，找出当前最可能阻塞项目推进的问题。
请区分：明确阻塞、潜在风险、需要确认的问题。
```

#### 生成本周计划

```text
请基于当前项目状态，为这个项目生成本周计划。
请限制在 3 个重点以内，并说明为什么这些事情最重要。
```

## 16. 成功指标

### 16.1 产品使用指标

- 每个项目平均上下文数量
- 每个项目平均任务数量
- 每个项目 AI Run 数量
- 每周活跃项目数
- 每周活跃用户数
- AI 输出被保存、转任务或转决策的比例

### 16.2 业务价值指标

重点观察以下问题：

- 团队是否更少翻聊天记录？
- 项目成员是否更快理解项目状态？
- AI 是否能更快进入项目上下文？
- 项目下一步是否更清晰？
- 周会或同步会是否变短？

### 16.3 MVP 验证标准

如果满足以下条件，说明 MVP 方向成立：

- 团队每天都会打开项目驾驶舱。
- 至少 70% 的活跃项目有任务和上下文记录。
- AI Run 输出中有至少 30% 被转化为任务、决策或上下文。
- 团队成员认为它比飞书多维表格/文档更适合推进小项目。

## 20. 初版产品原则

1. **项目优先，不是聊天优先**  
   所有信息围绕项目组织，而不是围绕会话组织。

2. **上下文优先，不是文档优先**  
   文档只是上下文的一种，真正重要的是 AI 和人都能理解项目背景。

3. **小团队优先，不服务复杂组织**  
   不为了大企业管理需求牺牲小团队速度。

4. **AI 输出必须可沉淀**  
   每次 AI 生成的有价值内容，都应该能转为任务、决策或上下文。

5. **下一步必须清楚**  
   用户打开任何项目，都应该知道接下来最重要的事情是什么。

6. **默认轻量，不做重型流程**  
   不做复杂审批、甘特图、权限和项目模板。
